# Уроки разработки и эталонный пример

Этот файл — «почему» пайплайна. Он не вводит новых правил, а показывает, из каких реальных
провалов на тестах выросло каждое правило, и даёт один эталонный первый ответ. Полезен, когда
модель/исполнитель «прочитал, но не понял зачем» — здесь видно цену нарушения.

## Часть A. Карта «провал → правило»

Документ-предшественник (v2.9) разрастался реактивно: каждый тест на новой модели вскрывал
новый способ сломать продакшен, и появлялось новое правило. Ниже — этот лог в сжатом виде.
Используй таблицу как быстрый self-check перед отправкой ответа: если узнаёшь свой ответ в
левой колонке — сработало антипаттерн-правило справа.

| Что реально случилось на тесте | Правило-противоядие | Где живёт |
|---|---|---|
| Промпт написан без указания модели/инструмента | Model Block обязателен перед каждым промптом | SKILL §2 |
| Все персонажи слиты в один промпт → drift и смешение | Отдельный Character Board на каждого важного героя; общий — только Cast Lineup | manual §7 |
| Выбрали Kling для сложной логики, где нужен Seedance | Модель выбирается по задаче кадра, не по названию | manual §10 |
| Видео-промпт без таймлайна и без ссылок на panels | Timecoded Prompt обязан ссылаться на Shot Board panels по секундам | SKILL §5 |
| Keyframes делались как обязательный этап после раскадровки | Раскадровка — главный управляющий документ; keyframe — опциональный якорь | SKILL §4 |
| Одну красивую картинку выдали за раскадровку (Single Frame Trap) | Раскадровка = ≥4 panels, порядок, timing, камера, переходы | SKILL §4 / manual §8 |
| Борды создали, но в финальных промптах не упомянули | Board Utilization: каждый борд используется через Reference Map | SKILL §6 |
| Портрет назвали «keyframe» и пытались анимировать failed-кадр | Failed keyframe = провал гейта; нельзя анимировать, сначала исправить | manual §18 |
| Заявили «Soul ID создан», хотя training провалился/подменён | Identity honesty: честные статусы training | manual §18 |
| Написали «готово» без файла, fps, QC | Delivery Evidence обязателен для статуса «готово» | SKILL §8 |
| Утверждали доступность инструмента без проверки окружения | Tool Availability Check; этап не пропускать | manual §18 |
| Запихнули 7 beats в 10 секунд → развал генерации | Density Control: считать beats, дробить на клипы | SKILL §7 |
| Instant сразу генерил картинку; даже обычный Thinking иногда; следовал только Extended Thinking | Safe Mode + First Response Contract + Anti-Instant guard | SKILL §0 / templates §14 |
| Быстрая модель «говорит концептами», а не действиями | Concept → Action: каждый концепт → deliverable + gate | SKILL §10 |
| Выдали «Production Log (старт)» как будто это готовый план | Production Log — журнал решений, не замена пайплайна и не «сделано» | manual §21 |
| Английские статусы (PLAN ONLY) без русского пояснения | Русский слой ответа | manual §24 |
| Кириллицу/логотипы генерировали прямо в видео | Текст/бренды → чистый кадр + post | SKILL §9 |
| Продолжили молча после failed/rejected generation | Failure Gate: остановиться, описать, предложить стратегию | manual §18 |

**Главный мета-урок всей истории:** прочитать пайплайн ≠ следовать ему в момент исполнения.
Особенно у моделей с доступом к инструментам (агент, подключённый к платформе) — они реально
исполняют и потому умеют «галлюцинировать выполнение»: называть черновик финалом, ошибку —
правильным workflow, незавершённый training — готовым. Поэтому слой дисциплины (SKILL §8,
manual §18) важнее, чем любой отдельный промпт.

**Второй мета-урок:** один и тот же документ даёт РАЗНЫЙ результат на разных режимах модели.
Лечится не удлинением документа, а коротким верхним control-промптом (Safe Mode / PLAN ONLY)
и контрактом первого ответа. Длинный документ — база знаний; короткий control-промпт — команда
исполнения. Для продакшена нужны оба.

## Часть B. Эталонный первый ответ (worked example)

Канонический тест-кейс из истории разработки — «короткий мультфильм про драконов-скейтеров в
хип-хоп эстетике». Вот как должен выглядеть **первый** ответ по контракту (статус «только план»,
без генерации). Это образец тона, объёма и структуры — не шаблон для копирования дословно.

```text
Статус: только план — без генерации.

1. Что я понял
Короткий стилизованный мультфильм (~30–45 c) про драконов-скейтеров, эстетика хип-хоп/стрит-арт.
Трюки — основное действие. Уникальный стиль, готовых референсов нет.

2. Тип задачи
Стилизованный action-мульт, musical-driven short. Сложность высокая (уровень 3–4).

3. Сложность и риски
- character drift между шотами;
- сложная анатомия (дракон + скейт-пропорции);
- сложные трюки = тяжёлое движение;
- generic fantasy / похожесть на Spider-Verse;
- слишком много beats в одном клипе.

4. Креативные варианты
A — безопасный: плоский граффити-2D, жирные контуры, мало деформаций.
B — сбалансированный: стилизованный 3D, неоновый ночной скейт-парк, теги на стенах.
C — амбициозный: гибрид «живые татуировки/стрит-арт», резкий кадр-в-кадр монтаж.

5. Рекомендованный вариант
B — сбалансированный: уникальный, но управляемый; реализуем без экстремального риска по drift.

6. Производственный путь
Если Higgsfield: борды (GPT Image 2 / Nano Banana Pro) → раскадровка → Segment Storyboards →
Seedance 2.0 Fast (тест логики) → Seedance 2.0 (финал по сегментам) → монтаж.
Если не Higgsfield: те же борды на GPT Image 2 / Flux / Midjourney → Sora / Veo / Kling по сегментам.

7. Нужные борды
- Борд стиля; Борд культуры мира (субкультура); Борд главного дракона; Борд скейтбордов;
  Борд локации; Борд движения (Motion Philosophy); Главная раскадровка; Раскадровки сегментов;
  Схемы трюков (Action Boards).

8. Нужны ли:
Главная раскадровка — да. Раскадровки сегментов — да (ролик сложный, несколько генераций).
Схемы движения — да (трюки). Доп. визуальные якоря — точечно (финальный кадр трюка, packshot).

9. Контрольные точки
Стиль → персонаж → мир → раскадровка → motion-тест → финал по сегментам → монтаж.

10. Первый конкретный шаг
Этап 1 — Борд стиля (Style Board).
Модель: GPT Image 2 (или Nano Banana Pro через Higgsfield).
Цель: зафиксировать визуальный язык: палитра (ночь/неон/золото/чёрный/кислотный),
фактуры, силуэт дракона, camera/motion language, do/don't. 8–12 панелей.
Результат: style_board_v01.png.

Останавливаюсь здесь и жду подтверждения.
```

Чем этот ответ правильный: статус проставлен; media не генерируется; даны варианты, а не один
навязанный; борды перечислены; раскадровка — primary, keyframes сведены к точечным якорям;
есть и Higgsfield-, и не-Higgsfield-путь; первый шаг конкретен (модель + цель + deliverable);
ответ останавливается и ждёт подтверждения.

**Анти-пример** (так делать нельзя): сразу выдать `Create a high-resolution character board of
a dragon skater...` или сгенерировать картинку. Это нарушает контракт первого ответа и, по
истории тестов, ровно то, что делали Instant-режимы.
